PayGrid Developer API (`/api/v1/`). API key auth only. Dashboard handles keys, reports, and ledger export. Rate limit: 60 requests/minute per API key.

## 1. API key

[Dashboard](https://paygrid.briq.tz) → generate `sk_test_...` or `sk_live_...`.

<Note>
  Test keys (`sk_test_*`) never move real money. Creating a payment with a test key returns `403 TEST_MODE_UNAVAILABLE` unless test mode is enabled on the platform — when it is, test payments run in a sandbox and never touch real balances. Payouts are always blocked for test keys; read endpoints work normally.
</Note>

## 2. Ping

```http
Authorization: Bearer <your_api_key>
```

```bash
curl -X GET "https://meet.briq.tz/api/v1/ping" \
  -H "Authorization: Bearer YOUR_API_KEY"
```

## 3. Create payment

`Idempotency-Key` required — missing it returns `400 IDEMPOTENCY_KEY_REQUIRED`. [Idempotency](/guides/idempotency).

Minimum amount: **500 TZS** for TZS mobile payments.

Include `webhook_url` on the request (or set [merchant defaults](/guides/merchants/merchant-default-urls)). PayGrid posts `payment.{status}` events on status transitions as the payment moves from `pending` to a terminal status (`completed`, `failed`, `cancelled`, or `expired` after ~30 minutes). There is no webhook at creation time. See [Webhooks](/guides/webhooks) and [Payment status](/guides/payments/status).

```bash
curl -X POST "https://meet.briq.tz/api/v1/payments" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: $(uuidgen)" \
  -d '{"amount":5000,"currency":"TZS","type":"mobile","phone":"255712345678","customer":{"firstname":"John","lastname":"Doe","email":"john@example.com"},"reference":"ORDER_123","webhook_url":"https://api.example.com/paygrid/webhooks"}'
```

Returns `201` with `status: "pending"` — the customer gets a USSD prompt on their phone and authorizes with their wallet PIN, typically within seconds.

## 4. Balance

```bash
curl -X GET "https://meet.briq.tz/api/v1/merchants/me/balance" \
  -H "Authorization: Bearer YOUR_API_KEY"
```

## Networks

`network` is optional — the operator is auto-detected from the phone number. When provided, values are case-insensitive:

| Network | Mobile money service |
| :--- | :--- |
| vodacom | M-Pesa |
| tigo | Tigo Pesa |
| airtel | Airtel Money |
| halotel | Halopesa |
| ttcl | T-Pesa |

The aliases `mpesa` (→ vodacom) and `mixx` (→ tigo) are also accepted; any other value returns `400 VALIDATION_ERROR`.

Collections: `type: "mobile"` ([mobile payment](/guides/payments/create-mobile)), `type: "card"` ([card payment](/guides/payments/create-card)), and `type: "dynamic-qr"` ([dynamic QR](/guides/payments/create-dynamic-qr)) — all via `POST /api/v1/payments`.
