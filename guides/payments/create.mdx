`POST /api/v1/payments`

`Idempotency-Key` required — requests without it are rejected with `400 IDEMPOTENCY_KEY_REQUIRED`. [Idempotency](/guides/idempotency).

Returns `201` with the new payment, or `200` when replaying an existing `Idempotency-Key`.

## Body

| Field | Required | Notes |
| :--- | :--- | :--- |
| `amount`, `currency`, `type`, `customer` | yes | Currencies: `TZS`, `USD`, `KES`, `UGX`. Minimum **500 TZS** for TZS mobile |
| `phone` | yes | All payment types. See formats below |
| `network` | no | Auto-detected from the phone number; validated when provided |
| `reference`, `metadata` | no | Duplicate live `reference` → `409 DUPLICATE_REFERENCE` (see below) |
| `webhook_url` | no | Status events; overrides merchant default |
| `callback_url` | no | Terminal status only; overrides merchant default |
| `redirect_url`, `cancel_url` | card | Required for `type: "card"` (hosted checkout return URLs) |

`customer` requires `firstname`, `lastname`, `email`. Card payments additionally require `address`, `city`, `state`, `postcode`, `country`.

## Phone formats

For TZS, all of these normalize to `255712345678`: `0712345678`, `712345678`, `255712345678`, `+255712345678`. After normalization the number must be a valid Tanzanian mobile (`255` + `6`/`7` + 8 digits) — anything else returns `400 VALIDATION_ERROR`.

## Networks

`network` is **optional** — Selcom detects the operator from the phone number. When you do send it, use lowercase:

| `network` | Provider |
| :--- | :--- |
| vodacom | M-Pesa (alias: `mpesa`) |
| tigo | Tigo Pesa / Mixx (alias: `mixx`) |
| airtel | Airtel Money |
| halotel | Halopesa |
| ttcl | T-Pesa |

Invalid values return `400 VALIDATION_ERROR`.

## What the customer sees (mobile)

A USSD prompt pops up on the customer's phone within seconds. They enter their wallet PIN and the payment usually completes in seconds. If they decline, the payment moves to `failed`/`cancelled`; if they ignore the prompt, it stays `pending` and expires roughly 30 minutes after creation.

## Card and dynamic QR

`type: "card"` and `type: "dynamic-qr"` create a Selcom hosted checkout. The response includes a `payment_url` the customer opens to pay (card entry or QR scan); `dynamic-qr` also returns `qr_code`. Card requires `redirect_url`, `cancel_url`, and the full customer billing address.

## Error semantics

| HTTP | `error_code` | Meaning |
| :--- | :--- | :--- |
| 400 | `IDEMPOTENCY_KEY_REQUIRED` | Missing `Idempotency-Key` header |
| 400 | `VALIDATION_ERROR` | Invalid phone/network/currency, or amount below 500 TZS |
| 402 | `PAYMENT_DECLINED` | Provider declined synchronously — no charge; `details` has `transaction_id` and `status: "failed"` |
| 409 | `DUPLICATE_REFERENCE` | A live (`pending`/`processing`/`completed`) payment with this `reference` exists; `details` has `existing_transaction_id`, `existing_status`, `reference`. Reuse is fine after `failed`/`cancelled`/`expired` |
| 502 | `PROVIDER_UNAVAILABLE` / `PROVIDER_ERROR` | Provider outage or timeout — retry with the **same** `Idempotency-Key` |

Full list: [Errors](/guides/errors).

```bash
curl -X POST "https://meet.briq.tz/api/v1/payments" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: $(uuidgen)" \
  -d '{"amount":5000,"currency":"TZS","type":"mobile","phone":"255712345678","customer":{"firstname":"John","lastname":"Doe","email":"john@example.com"},"reference":"ORDER_123","webhook_url":"https://api.example.com/paygrid/webhooks"}'
```
